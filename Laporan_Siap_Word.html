1. Pendahuluan, Kebutuhan, dan Aturan Bisnis

1.1 Latar Belakang Masalah

Dalam lingkungan pendidikan tinggi, khususnya pada institusi yang berfokus pada teknologi dan inovasi seperti Institut Teknologi Sepuluh Nopember (ITS), kolaborasi lintas disiplin ilmu merupakan sebuah keharusan. Mahasiswa seringkali dituntut untuk membentuk tim yang solid guna mengikuti berbagai ajang perlombaan bergengsi (seperti PKM, Gemastik), membangun proyek rintisan (startup), hingga menyelesaikan tugas akhir komprehensif. Sebuah tim yang ideal biasanya membutuhkan perpaduan berbagai keahlian (skills), mulai dari software engineering, UI/UX design, data analysis, hingga business management.

Namun, pada praktiknya, proses pencarian anggota tim ini masih sering menghadapi kendala yang signifikan. Pencarian kandidat umumnya masih dilakukan secara konvensional dan sporadis melalui grup-grup obrolan (WhatsApp, Line). Metode ini dinilai sangat tidak efisien karena informasi rekrutmen mudah tenggelam, sulitnya melakukan validasi portofolio atau keahlian calon anggota secara objektif, serta tingginya kemungkinan terjadinya ketidakcocokan peran (role mismatch).

Berangkat dari permasalahan tersebut, dirancanglah sebuah sistem informasi bernama TeamUp. Sistem informasi ini ditujukan sebagai platform terpusat yang mewadahi proses matchmaking dan rekrutmen internal mahasiswa. Untuk mendukung jalannya aplikasi tersebut secara real-time dan efisien, diperlukan sebuah rancangan basis data (database) relasional yang tangguh, terstruktur, dan terbebas dari anomali data. Laporan ini akan membahas secara komprehensif mengenai perancangan arsitektur, implementasi, hingga optimasi dari database sistem TeamUp.

1.2 Aturan Bisnis (Business Rules)

Untuk memastikan integritas data dan kesesuaian dengan proses bisnis (business logic) di dunia nyata, arsitektur database dibangun berdasarkan aturan-aturan berikut: 1. Otentikasi dan Eksklusivitas Ruang Lingkup: Sistem ini ditujukan secara eksklusif untuk ruang lingkup mahasiswa aktif ITS. Oleh karena itu, diterapkan konstrain validasi pada entitas `USER` di mana hanya pendaftaran menggunakan surat elektronik (email) dengan domain `@student.its.ac.id` yang akan diterima oleh sistem. 2. Kardinalitas Rekrutmen (1:N): Seorang pengguna (User) bertindak sebagai inisiator proyek dan memiliki hak untuk membuat lebih dari satu postingan rekrutmen (Recruitment Post). Sebaliknya, setiap postingan hanya dapat dimiliki oleh satu pengguna utama. 3. Spesifikasi Peran/Posisi (1:N): Dalam satu payung postingan rekrutmen, inisiator dapat membuka berbagai posisi (Position) yang spesifik (misalnya: 1 posisi Frontend, 1 posisi UI/UX). 4. Relasi Keahlian Majemuk (M:N): Sistem mengadopsi fleksibilitas keahlian, di mana satu posisi dapat mensyaratkan banyak keahlian (Skill), dan satu entitas keahlian tunggal dapat menjadi prasyarat untuk berbagai posisi berbeda. Fleksibilitas yang sama juga diterapkan pada kepemilikan keahlian oleh User. 5. Kebijakan Anti-Spam pada Pendaftaran: Mahasiswa diperbolehkan melamar ke berbagai posisi di berbagai proyek. Namun, sistem database secara ketat menolak duplikasi pendaftaran (seorang mahasiswa melamar pada posisi ID yang persis sama lebih dari satu kali). 6. Otomatisasi Status Berbasis Kuota: Integritas proses rekrutmen dijaga menggunakan sistem pemicu otomatis (Trigger). Apabila sebuah posisi memiliki kuota 2 orang, dan inisiator telah mengubah status 2 pelamar menjadi 'Diterima', maka sistem database akan secara otomatis mengunci posisi tersebut dengan mengubah statusnya menjadi 'Tutup'.

3. Skema Relasional, Constraint, dan Normalisasi

3.1 Skema Relasional (PDM)

Dalam mentranslasikan desain konseptual menjadi Physical Data Model (PDM), relasi Many-to-Many (M:N) yang terdapat pada desain konseptual dipecah (didekomposisi) menggunakan tabel perantara (junction table). Hal ini menghasilkan dua tabel fisik tambahan, yaitu `user_skills` dan `position_skills`, sehingga total tabel fisik menjadi 9 tabel.

3.2 Constraint Integritas

Database diimplementasikan dengan pengamanan level basis data (database constraints). Primary Key menggunakan tipe data `INT` dengan sifat `AUTO_INCREMENT`. Sementara itu, integritas referensial antar tabel dijaga menggunakan Foreign Key dengan klausa `ON DELETE CASCADE`. Pendekatan kaskade ini memastikan bahwa apabila sebuah entitas induk dihapus (contoh: akun User dihapus), maka seluruh data turunan yang bergantung padanya (portofolio, lamaran, postingan) akan ikut terhapus secara otomatis guna mencegah terjadinya orphan data (data tak bertuan) yang membebani kapasitas penyimpanan.

3.3 Analisis Normalisasi

Untuk menghindari anomali insert, update, dan delete, serta mengurangi redudansi data, skema database telah dirancang dan divalidasi hingga memenuhi standar Bentuk Normal Ketiga (3NF): - 1NF (First Normal Form): Seluruh tabel telah dipastikan hanya memiliki atribut atomik tunggal. Tidak ada penggunaan array atau koma untuk menyimpan nilai majemuk. Sebagai contoh nyata, keahlian pengguna tidak disimpan sebagai teks `(Programming, Design, Marketing)` di dalam tabel `USER`, melainkan diisolasi pada entitas tersendiri dan dihubungkan. - 2NF (Second Normal Form): Setiap tabel telah memiliki Primary Key absolut. Seluruh atribut yang bukan kunci bergantung penuh secara fungsional pada keseluruhan Primary Key. Pada tabel junction `position_skills` (dengan Primary Key komposit `id_pos` dan `id_skill`), tidak ada atribut sekunder yang hanya bergantung secara parsial pada salah satu kunci pokok. - 3NF (Third Normal Form): Database dinyatakan bebas dari dependensi transitif. Artinya, setiap atribut non-kunci hanya bergantung secara langsung kepada Primary Key utama tabel tersebut, dan tidak bergantung pada atribut non-kunci lainnya.

4. Dokumentasi Implementasi dan Data Uji

4.1 Implementasi DDL (Data Definition Language)

Tahap implementasi dilakukan menggunakan Relational Database Management System (RDBMS) MySQL. Perintah Data Definition Language (DDL) dieksekusi secara berurutan sesuai dengan hierarki dependensi Foreign Key, dimulai dari tabel independen hingga tabel dependen.

Implementasi Kode:

-- Membuat Database
CREATE DATABASE IF NOT EXISTS teamup_db;
USE teamup_db;


-- Tabel Master: users
CREATE TABLE users (
    id_user INT AUTO_INCREMENT PRIMARY KEY,
    nama VARCHAR(255) NOT NULL,
    email VARCHAR(255) NOT NULL UNIQUE,
    password VARCHAR(255) NOT NULL,
    departemen VARCHAR(100) NOT NULL,
    angkatan INT NOT NULL,
    bio TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);


-- Tabel Master: skills
CREATE TABLE skills (
    id_skill INT AUTO_INCREMENT PRIMARY KEY,
    nama_skill VARCHAR(100) NOT NULL UNIQUE,
    kategori VARCHAR(50) NOT NULL
);


-- Tabel Transaksional: portofolios
CREATE TABLE portofolios (
    id_portofolio INT AUTO_INCREMENT PRIMARY KEY,
    id_user INT NOT NULL,
    judul VARCHAR(255) NOT NULL,
    deskripsi TEXT,
    link VARCHAR(255),
    tgl_dibuat DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (id_user) REFERENCES users(id_user) ON DELETE CASCADE
);


-- Tabel Transaksional: recruitment_posts
CREATE TABLE recruitment_posts (
    id_post INT AUTO_INCREMENT PRIMARY KEY,
    id_user INT NOT NULL,
    judul VARCHAR(255) NOT NULL,
    deskripsi TEXT NOT NULL,
    status ENUM('Buka', 'Tutup') DEFAULT 'Buka',
    tgl_dibuat DATETIME DEFAULT CURRENT_TIMESTAMP,
    tgl_tutup DATETIME NOT NULL,
    FOREIGN KEY (id_user) REFERENCES users(id_user) ON DELETE CASCADE
);


-- Tabel Transaksional: positions
CREATE TABLE positions (
    id_pos INT AUTO_INCREMENT PRIMARY KEY,
    id_post INT NOT NULL,
    nama_pos VARCHAR(100) NOT NULL,
    kuota INT NOT NULL,
    status ENUM('Buka', 'Tutup') DEFAULT 'Buka',
    FOREIGN KEY (id_post) REFERENCES recruitment_posts(id_post) ON DELETE CASCADE
);


-- Tabel Junction (M:N): user_skills
CREATE TABLE user_skills (
    id_user INT NOT NULL,
    id_skill INT NOT NULL,
    PRIMARY KEY (id_user, id_skill),
    FOREIGN KEY (id_user) REFERENCES users(id_user) ON DELETE CASCADE,
    FOREIGN KEY (id_skill) REFERENCES skills(id_skill) ON DELETE CASCADE
);


-- Tabel Junction (M:N): position_skills
CREATE TABLE position_skills (
    id_pos INT NOT NULL,
    id_skill INT NOT NULL,
    PRIMARY KEY (id_pos, id_skill),
    FOREIGN KEY (id_pos) REFERENCES positions(id_pos) ON DELETE CASCADE,
    FOREIGN KEY (id_skill) REFERENCES skills(id_skill) ON DELETE CASCADE
);


-- Tabel Transaksional: applications
CREATE TABLE applications (
    id_app INT AUTO_INCREMENT PRIMARY KEY,
    id_pos INT NOT NULL,
    id_user INT NOT NULL,
    tgl_daftar DATETIME DEFAULT CURRENT_TIMESTAMP,
    status ENUM('Menunggu', 'Diterima', 'Ditolak') DEFAULT 'Menunggu',
    FOREIGN KEY (id_pos) REFERENCES positions(id_pos) ON DELETE CASCADE,
    FOREIGN KEY (id_user) REFERENCES users(id_user) ON DELETE CASCADE
);


-- Tabel Transaksional: notifikasi
CREATE TABLE notifikasi (
    id_notifikasi INT AUTO_INCREMENT PRIMARY KEY,
    id_user INT NOT NULL,
    judul VARCHAR(255) NOT NULL,
    pesan TEXT NOT NULL,
    kategori VARCHAR(50) NOT NULL,
    id_terkait INT,
    status_baca BOOLEAN DEFAULT FALSE,
    tgl_dibuat DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (id_user) REFERENCES users(id_user) ON DELETE CASCADE
);

8.2 Kesimpulan

Pengembangan arsitektur basis data relasional untuk platform TeamUp dapat dikonklusikan berhasil secara komprehensif.
